# 产品定位与边界

## 产品名称

TokenHub

## 产品定位

面向企业私有化部署的 AI API Gateway 与 Token 治理平台，统一管理多模型接入、内部 API Key、额度分配、调用审计、成本分析和模型路由，为企业构建安全、可控、可追踪的 AI 使用入口。

## 目标客户

| 客户类型 | 典型诉求 |
| --- | --- |
| 中大型企业 | 多部门、多项目共用模型能力，需要统一预算、权限、审计和报表 |
| 政企与工业客户 | 内网部署、审计留痕、敏感数据脱敏、供应商可控 |
| AI 应用团队 | 希望用一个统一 API 接入多个模型，减少 Provider 差异 |
| 平台工程团队 | 需要建设企业内部 AI 基础设施，向业务团队发放可治理的 API Key |
| 财务与管理团队 | 需要了解模型使用量、成本归属、异常消耗和预算执行情况 |

## 核心用户角色

| 角色 | 关注点 |
| --- | --- |
| 平台管理员 | Provider、模型、全局策略、系统运行状态 |
| 项目管理员 | 项目 Key、额度、模型白名单、项目成本 |
| 开发者 | 稳定的 API、清晰的错误、低迁移成本 |
| 安全管理员 | 审计日志、敏感数据、异常调用、访问控制 |
| 财务/运营 | 成本归因、趋势分析、预算告警 |

## 核心场景

1. 企业内部统一模型出口
   - 所有应用调用 TokenHub，不直接持有上游 Provider Key。

2. 多模型统一接入
   - 使用统一 API 接入 OpenAI、Azure OpenAI、Claude、Gemini、DeepSeek、Qwen、本地 vLLM/Ollama。

3. 项目级 API Key 管理
   - 每个项目独立 Key、独立模型白名单、独立预算和独立审计。

4. 成本与额度治理
   - 对项目、用户、Key、模型设置日/月额度，超额自动拒绝或降级。

5. 模型路由与容灾
   - 按成本、延迟、可用性、区域和优先级选择 Provider，失败时按策略重试或降级。

6. 审计与安全
   - 对请求元信息、用量、成本、异常行为和策略命中进行留痕。

## 差异化价值

| 方向 | TokenHub 的做法 |
| --- | --- |
| 企业定位 | 强调私有化、审计、权限、成本治理和稳定性 |
| 模型接入 | 多 Provider Adapter，屏蔽协议和计费差异 |
| 内部治理 | Project、Team、User、Key、Quota、Policy 形成闭环 |
| 成本管理 | 从 Token 统计升级到预算、成本归因、告警和报表 |
| 安全合规 | 密钥托管、敏感数据脱敏、审计日志、企业身份源 |
| 部署方式 | Docker、Helm、内网、离线、国产化环境预留 |

## 非目标

以下能力不作为 TokenHub 的产品方向：

- 不提供非授权账号账号的多人共享和转 API 分发。
- 不以规避上游服务条款、风控或计费规则作为产品卖点。
- 不承诺规避、绕开或替代上游 Provider 的计费规则。
- 不复制第三方项目代码、SQL、前端组件、接口实现或配置结构。
- 不在 MVP 阶段实现完整支付系统、代理池、账号池和面向个人用户的自助充值。

## 合规边界

TokenHub 只应该连接企业合法持有的 Provider 凭证，包括：

- 企业官方 API Key。
- 企业云账号下的 Azure OpenAI、Vertex AI、Bedrock 等实例。
- 企业采购或授权的模型服务。
- 企业内网部署的 vLLM、Ollama、TGI、LMDeploy 等本地推理服务。

平台应在 README、管理后台和部署文档中明确提示：上游模型使用必须遵守对应 Provider 的服务条款、企业安全制度和所在地区法律法规。

## 成功指标

| 指标 | MVP 目标 |
| --- | --- |
| API 兼容性 | 主流 OpenAI SDK 可以无改造接入 |
| 网关可用性 | 单实例稳定转发，支持流式响应 |
| 统计完整性 | 请求、Token、成本、错误、路由结果可追踪 |
| 管理效率 | 管理员可在后台完成项目、Key、额度、Provider 配置 |
| 安全可审计 | 关键操作与调用链路可查询、可导出、可留痕 |
